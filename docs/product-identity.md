# TriForge AI Product Identity

**Status:** Locked — Section 1 Foundation  
**Date:** 2026-04-06  
**Authority:** This document defines the canonical product direction for all future restructuring work.

---

## What TriForge Is

TriForge AI is a desktop AI council and remote computer operator that helps users think through work and carry out tasks directly inside their own apps, files, browser, and workflows.

It has two core functions:

1. **Think with the user** — multi-agent reasoning, planning, guidance, and decision support through Chat.
2. **Do work for the user** — operating on files, apps, browser workflows, and machine tasks with user control and approvals.

These two functions are not separate products. They are two sides of the same AI worker.

---

## User Promise

When I open TriForge, I should feel like I now have a serious AI worker on my machine — not a collection of unrelated tools.

---

## What TriForge Is Not

TriForge is no longer a broad dashboard of separate modes, experiments, and themed areas competing for attention.

It is not:

- A loose collection of tools with separate identities
- A theme-based or mode-based experience
- An experimental feature showcase
- A general productivity dashboard

Every screen, panel, and interaction must serve one of the two core functions: thinking with the user or doing work for the user.

---

## Opening Impression

When a user opens TriForge for the first time, or returns after time away, the immediate impression must be:

- **Focused** — one clear product, one clear purpose
- **Powerful** — capable of serious work, not toy-level assistance
- **Trusted** — deliberate, controlled, transparent in what it does
- **Ready to work** — the interface orients immediately around the user's next task, not around the product's own features

If the opening experience feels like a menu of options, it has failed. It should feel like a capable worker waiting for direction.

---

## Section 1 Scope Boundary

This document is the output of **Section 1, Goal 1** of the TriForge product restructuring plan.

Section 1 establishes product identity only. It does not:

- Define navigation structure
- Assign screens to functions
- Redesign UI layout
- Change any application code or behavior

All subsequent sections (screen audit, navigation restructuring, implementation) must treat this document as the fixed anchor. If a proposed change cannot be justified against the two core functions and the user promise above, it does not belong in TriForge.

---

## Locked Direction for Future Sections

All future restructuring work must answer these questions before proceeding:

1. Does this change serve "think with the user" or "do work for the user"?
2. Does it reinforce or undermine the opening impression of a focused, powerful, trusted AI worker?
3. Does it reduce or increase the sense of a scattered tool collection?

If a proposed screen, feature, or section cannot clearly answer Question 1, it should not exist in TriForge.

The product direction is locked. Navigation, screen assignments, and implementation work in later sections must follow this identity — not redefine it.

**Hard guardrails for all future work:**

- The product must always present itself as one system, not multiple modes or themed environments.
- No future feature may introduce a new top-level identity outside the five core pillars.

These guardrails apply to all sections, including Operate, Sessions, and any capability added after Section 1. If a proposed addition would require the user to mentally switch contexts or learn a separate product identity, it does not belong at the top level of TriForge.
